**1. Demo 简介**

* **是什么：** 一个基于 Vue 3 + FastAPI 的 Web 端 AI 智能面试助手应用，通过浏览器实时语音识别捕获面试官提问，结合火山引擎方舟大模型和双知识库架构（云端知识库 / 本地 ChromaDB 向量数据库），为求职者提供个性化的实时回答建议。核心理念是**把 AI 落实到真正的业务场景**——帮助面试者从"知识记忆"转向"能力展现"。

* **面向谁：** 正在求职、需要面试辅助的个人用户，以及希望提升面试表现、练习面试回答的求职者。

* **主要功能：**

  1. **实时语音识别**：基于浏览器 Web Speech API，自动识别面试官的提问，2秒停顿自动结束识别，支持错误自动重启。

  [截图：首页]

  2. **AI智能回答建议**：调用火山引擎方舟大模型（deepseek-v4-flash），结合个人知识库（简历/项目经历），基于 STAR 法则生成结构化回答建议，支持流式（SSE）打字机效果渲染。无知识库时自动降级为通用回答模式。

  [截图：面试中]

  3. **结构化对话展示**：左侧展示面试官问题，右侧展示 AI 生成的回答建议，支持展开/收起长回答、自动滚动到最新消息、时间戳显示。

  [截图：对话详情]

  4. **双知识库架构**：支持火山引擎云端知识库和本地 ChromaDB 向量数据库自由切换——云端适合快速接入，本地完全免费、支持离线、数据完全可控。使用 BGE-Large-ZH-v1.5 开源中文 Embedding 模型进行向量化。

  5. **文档上传与智能切片**：支持 PDF、Word、TXT 等多种格式文档上传，基于 Recursive Character Text Splitter 实现可配置的语义切片策略（chunk_size=500, chunk_overlap=50），检索结果返回前进行语义相关性校验，确保引入的知识与问题高度相关。

**2. Demo 创作思路**

* **灵感来源：** 作为一名技术面试官，我见过太多候选人因为紧张而无法充分展示自己的能力。很多时候他们并非没有能力，而是缺乏一个实时的"辅助大脑"来帮他们组织语言、回忆项目细节。这让我想到，能否用AI技术打造一个面试辅助工具，在面试过程中实时帮助求职者。

* **想解决的问题：** 
  - 面试紧张导致思路混乱，回答不够结构化
  - 无法及时回忆起自己的项目经历和技术细节
  - 传统面试准备方式（背诵）无法应对突发问题
  - 缺乏个性化的实时回答建议

* **为什么做这个方向：** 
  - AI技术的发展让实时辅助成为可能，大模型+知识库（RAG）的组合能够实现个性化回答
  - 相比通用面试技巧网站，本产品结合用户真实背景，回答更具针对性
  - 浏览器端部署，无需安装额外软件，降低使用门槛
  - 作为全栈工程师，这个项目能够充分展示我的技术栈整合能力和 AI 工程化实践
  - **更深层的思考**：我相信面试的核心价值在于考察思维能力、组织能力和问题定义能力——"能够精准提出问题、定义问题的人，往往比单纯回答问题的人更具价值"。本项目遵循 **蒸馏→沉淀→溶解** 的知识工程方法论，将面试问答逻辑蒸馏为结构化 Prompt 模板，将知识库检索沉淀为可复用的 `KnowledgeProvider` 协议接口，再将它们在实时对话场景中灵活融合

**4. TRAE 实践过程**

作为全栈工程师，我全程主导项目开发，TRAE 作为工程助手辅助完成代码实现。以下是关键开发过程：

**步骤一：架构设计与技术选型**

我首先进行了需求分析和架构设计：
- 前端：Vue 3 + Vite + Tailwind CSS，选择 Vue 3 Composition API 提升代码复用性
- 后端：Python + FastAPI，选择 FastAPI 因其高性能和自动文档生成能力
- 数据库：PostgreSQL + SQLAlchemy，用于存储对话记录
- AI服务：火山引擎方舟大模型 + 双知识库架构（RAG）
- 知识库方案：火山引擎云端知识库（快速接入） + 本地 ChromaDB 向量数据库（免费、离线、数据可控）
- 向量化模型：BGE-Large-ZH-v1.5 开源中文 Embedding
- 语音识别：浏览器 Web Speech API（零成本）+ 火山引擎ASR（备用方案）

**步骤二：前端界面开发**

在我的指导下，TRAE 完成了前端组件的开发：
- `HomePage.vue`：科技风格首页，包含浏览器兼容性检测、开始面试按钮、配置入口
- `InterviewPage.vue`：面试主页面，包含对话列表、录音状态指示器、底部控制栏
- `DialogueItem.vue`：对话项组件，支持展开/收起长回答、时间戳显示
- `ConfigModal.vue`：配置弹窗，管理 API Key、知识库类型切换、模型 ID

**步骤三：语音识别模块开发（核心难点）**

这是整个项目最具挑战性的部分：
- `useSpeech.ts`：实现语音识别 Composable，处理麦克风权限、实时识别、2秒停顿自动结束
- **核心问题**：浏览器 Web Speech API 的 `onend` 事件和手动停止会并发触发，导致重复调用 `recognition.start()`
- **解决方案**：引入三重保护机制——`isStarting` 互斥锁 + `generation` 计数器 + `isStopping` 标志，确保同一时间只有一个启动操作
- **优化**：添加错误自动重启机制，网络错误或无语音输入时自动恢复

**步骤四：后端服务开发**

我设计了完整的后端架构，TRAE 辅助实现：
- `main.py`：FastAPI 入口，包含 CORS 配置、请求日志中间件、全局异常处理
- `llm.py`：火山引擎方舟大模型调用服务，支持流式（SSE）和非流式两种模式
- `knowledge.py`：火山引擎知识库检索服务，使用 SignerV4 签名认证，支持自定义参数
- `prompt.py`：Prompt 拼接服务，支持知识库增强和无知识库降级方案
- `logger.py`：统一日志模块，短日志控制台输出（便于快速定位）+ 长日志文件存储（按天分割，保留30天）

**步骤五：本地知识库模块开发**

为降低使用成本、提升数据隐私，我主导设计并实现了本地知识库方案：
- 定义 `KnowledgeProvider` 协议接口：抽象知识库检索能力，实现云端/本地双实现无缝切换
- 基于 LangChain + ChromaDB 构建本地向量数据库，支持 BGE-Large-ZH-v1.5 向量化
- 实现文档上传 API：支持 PDF、Word、TXT、Markdown 格式解析
- 实现可配置的语义切片策略（Recursive Character Text Splitter），检索结果返回前进行语义相关性校验
- 通过工厂模式（`KnowledgeProviderFactory`）根据配置动态创建对应的知识库提供者

**步骤六：部署脚本开发**

为了方便用户一键启动，我设计并实现了 Docker Compose 部署方案：
- Docker 容器管理，自动启动 PostgreSQL 和 ChromaDB
- 端口冲突检测和自动清理（避免每次启动端口变化）
- 服务启动等待逻辑（使用 `pg_isready` 检查数据库就绪状态）

**开发过程中的关键挑战：**

1. **语音识别稳定性问题**：Web Speech API 在某些浏览器中不稳定，需要实现完整的错误处理和自动恢复机制
2. **双知识库架构设计**：需要抽象统一的 `KnowledgeProvider` 协议，让云端知识库和本地 ChromaDB 能够无缝切换，同时保证检索效果一致性
3. **知识库检索延迟**：RAG检索需要一定时间，需要在前端显示加载状态，同时优化 Prompt 拼接逻辑
4. **流式响应优化**：大模型调用采用 SSE 流式响应，需要处理前端实时渲染和自动滚动
5. **配置管理**：API Key 和知识库配置需要支持前端参数和后端默认值两种方式，优先级明确
