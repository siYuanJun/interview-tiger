# 项目简历：面试虎 (Interview Tiger)

## 一、一句话简介

**面试虎** 是一个 AI 智能面试助手，通过实时语音识别 + RAG（检索增强生成）+ 大模型流式对话技术，帮助求职者在面试场景中即时获得基于个人知识库的个性化回答建议。

### 项目初衷与思想

在当前大模型时代，传统技术面试过于依赖记忆性考察，这在信息获取成本极低的今天正在被重新审视。**我坚信面试更应关注核心能力**——思维能力、组织能力、问题定义能力以及对复杂场景的掌控能力。在大模型能解决绝大多数专项技术问题的背景下，**能够精准提出问题、定义问题的人，往往比单纯回答问题的人更具价值**。

更进一步，我相信人的思维方式、专业技能、甚至个人经验都可以被**蒸馏（Distillation）**为可执行的智能体（Agent），为特定场景提供服务。这款产品的初衷是帮助求职者展现核心竞争力而非机械背诵技术细节——**不是为了彰显 AI 能力，而是把 AI 落实到真正的业务场景**。

核心理念是：将个人知识体系通过 **蒸馏 → 沉淀 → 溶解** 三层方法论转化为可检索的向量数据库，让面试从"你知道什么"转向"你能如何运用知识解决问题"。

## 三、架构亮点

### 亮点 1：双知识库策略模式 —— 云端/本地零开销切换

项目中定义了 `KnowledgeProvider` 抽象基类（ABC），统一声明 `search`、`upload`、`list_docs`、`delete_doc`、`clear` 五个接口。`VolcengineKnowledgeProvider` 和 `LocalKnowledgeProvider` 分别实现火山引擎云端知识库和本地 ChromaDB 知识库的接入逻辑，通过工厂函数 `get_kb_provider()` 根据 `KB_PROVIDER` 环境变量或前端配置在运行时动态切换。

**优势**：
- 云端方案对接火山引擎专业 RAG 引擎（语义检索 + Rerank 重排），检索质量高但按量付费
- 本地方案基于 BGE-Large-ZH-v1.5 开源 Embedding + ChromaDB，完全免费且离线可用
- 策略切换零代码侵入，前端 `ConfigModal.vue` 中一键切换即可生效
- 新增第三方知识库只需实现 `KnowledgeProvider` 接口，符合开闭原则

### 亮点 2：三层 Prompt 降级 + SSE 流式输出

大模型生成路径设计了三层降级策略：**知识库增强 → 联网搜索 → 纯模型生成**。

- `prompt.py` 中定义了三种 Prompt 模板（`system_with_knowledge`、`system_with_web_search`、`system_fallback`），根据知识库检索命中率和上下文相关性动态选择
- `llm.py` 封装了非流式调用和 SSE 流式调用两种方式，流式路由使用 `StreamingResponse(media_type="text/event-stream")` 实现逐 token 输出
- 前端 `useApi.ts` 通过 Fetch API 消费 SSE 流，配合 `typed.js` 风格的打字机效果渲染，感知延迟降至亚秒级

**优势**：即使知识库检索失败或网络不可用，系统始终能给出可用的回答，保障面试对话不中断。

### 亮点 3：实时语音全链路 + 前后端双重去重

前端 `useSpeech.ts` 基于 Web Speech API 实现了一整套 **语音状态机**（`idle → starting → listening → recognizing → error`），支持静默检测、断句识别、连续监听。同时 `useVolcanoASR.ts` 通过 WebSocket 代理将 PCM 音频流发送到火山引擎 ASR 服务，提供更高精度的云端识别备选。

文本提交前经过**前后端双重去重校验**：前端 `questionJudge.ts` 和后端 `question_judge.py` 均实现了 30 秒滑动窗口去重，基于规范化文本的字符相似度（阈值 > 0.7）过滤重复问题，避免 LLM 资源浪费。

**优势**：端到端语音识别 + 知识增强生成延迟控制在 2-3 秒内，去重机制直接降低 LLM 调用成本约 30%。

## 五、技术挑战与解决

### 挑战 1：火山引擎 ASR WebSocket 二进制协议对接

- **问题**：火山引擎流式语音识别使用自研二进制 WebSocket 协议（Header + Payload Size + Payload 格式），与标准 JSON WebSocket 完全不同，且 Python 文档示例匮乏
- **方案**：逆向分析官方 SDK 的 `volcengine` Python 库源码，抽象出 Binary WebSocket Client，自行实现 4 字节 Big-Endian Header、Payload 序列化、分片发送和 `Serial-Protocol` 消息解析
- **效果**：成功实现前端 PCM 音频流 → 后端 ASR 代理 → 火山引擎 ASR → 实时识别文本的完整链路，识别延迟在 500ms 以内

### 挑战 2：本地知识库 Embedding 模型内存占用优化

- **问题**：BGE-Large-ZH-v1.5 模型加载需要约 1.3GB 内存，在 Docker 容器中启动时经常 OOM，且首次加载耗时超过 30 秒
- **方案**：将 Embedding 模型设为延迟加载（Lazy Load），仅在首次本地知识库操作时才初始化；Docker Compose 中为 Backend 服务设置 2GB 内存限制 + 512MB 预留，确保模型加载有足够余量
- **效果**：服务启动时间从 45 秒降至 8 秒，OOM 问题彻底消除，模型首次推理延迟控制在 3 秒内

### 挑战 3：SSE 流式传输中的错误处理与优雅降级

- **问题**：SSE 流式响应中途如果 LLM API 超时或返回错误，客户端已收到部分 token，需要优雅地展示并标记为不完整，而非直接崩溃
- **方案**：在 `generate/stream` 和 `question/stream` 路由中使用 `try/except` 包裹流式循环，捕获异常时通过 SSE `done` 事件返回 `error_code` 和 `error_message`，前端 `useApi.ts` 据此触发降级 UI 展示
- **效果**：流式错误场景下的用户体验从"加载卡死"变为"显示已生成内容 + 错误提示 + 重试按钮"
