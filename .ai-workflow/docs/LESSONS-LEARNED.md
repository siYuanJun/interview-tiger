## 💡 有效模式

### 1. 规则引擎优先 + LLM兜底的问题判断方案

**模式**: 问题判断优先使用规则引擎（标点、疑问词、句式、长度四重规则），响应快且无API成本。当规则引擎无法确定时，才调用后端LLM进行二分类。

**适用场景**: 需要快速判断且有明确规则的场景

**收益**: 
- 响应时间 < 100ms（规则引擎）vs > 1s（LLM）
- 零API调用成本
- 可解释性强

### 2. Web Speech API + 火山引擎ASR双模式

**模式**: 主方案使用浏览器原生Web Speech API（零成本），备用方案使用火山引擎ASR（更高准确率但有成本）。用户可根据需要切换。

**适用场景**: 需要平衡成本和准确率的语音识别场景

**收益**: 
- 个人用户零成本使用
- 企业用户可选择更高准确率

### 3. 前端参数 > .env > 默认值的配置优先级

**模式**: API Key配置优先级为：前端请求参数 > .env配置 > 默认空值。这样既支持前端动态配置，又支持后端默认配置。

**适用场景**: 需要灵活配置外部服务API Key的场景

**收益**: 
- 用户可在前端直接配置，无需修改代码
- 开发环境可通过.env配置默认值
- 生产环境可通过环境变量配置

### 4. SSE流式响应 + 打字机效果

**模式**: 使用SSE（Server-Sent Events）实现流式响应，前端配合打字机效果逐字渲染，减少用户等待焦虑。

**适用场景**: 需要实时展示AI生成结果的场景

**收益**: 
- 用户感知延迟显著降低
- 交互体验更好

## 🚫 踩过的坑

### 1. API Key硬编码问题

**问题**: 初期API Key通过前端参数传递，但后端路由没有使用.env配置作为默认值，导致用户必须在前端输入API Key。

**严重程度**: 🟡中

**根因**: 后端路由直接使用前端传入的参数，没有实现默认值回退逻辑。

**方案**: 在后端路由中添加 `resolve_config()` 函数，实现前端参数 > .env > 默认空值的优先级逻辑。

**影响面**: 所有需要API Key的路由（question.py, generate.py, search.py）

**预防措施**: 所有外部服务调用的配置参数都应实现多层回退机制。

### 2. Web Speech API兼容性问题

**问题**: Web Speech API在部分浏览器（如Safari）支持不完整，导致语音识别功能不可用。

**严重程度**: 🟡中

**根因**: 不同浏览器对Web Speech API的支持程度不同。

**方案**: 添加浏览器兼容性检测，不支持时显示友好提示，并提供火山引擎ASR作为备用方案。

**影响面**: 语音识别模块

**预防措施**: 使用Web API前必须进行兼容性检测。

### 3. 缺少模块交付文档

**问题**: 项目完成后缺少模块交付文档（流程文档、设计文档、测试报告等），不符合ai-dev-workflow规范。

**严重程度**: 🟡中

**根因**: 开发过程中跳过了阶段5（验证）和阶段6（闭环交付）。

**方案**: 补全所有模块的交付文档，包括流程文档、设计文档、双角色摘要、验收检查清单和测试报告。

**影响面**: 全部6个模块

**预防措施**: 严格按照ai-dev-workflow的7个阶段执行，完成每个阶段的产出物。

## ⚠️ 风险预警

### 1. API调用成本

**风险**: 大模型调用和知识库检索都有API调用费用，长期使用可能产生较高成本。

**级别**: 🟡中

**缓解措施**: 
- 添加使用量监控
- 实现预算控制
- 提供成本估算页面

### 2. 浏览器兼容性

**风险**: Web Speech API和MediaRecorder API在部分浏览器不支持或支持有限。

**级别**: 🟡中

**缓解措施**: 
- 已添加兼容性检测
- 提供备用方案（火山引擎ASR）
- 在首页显示支持的浏览器列表

### 3. 大量对话性能

**风险**: 当对话数量较多时，前端渲染性能可能下降。

**级别**: 🟢低

**缓解措施**: 
- 实现虚拟滚动
- 或分页加载历史对话

### 4. 无数据库持久化

**风险**: 当前对话记录仅存储在内存中，刷新页面后丢失。

**级别**: 🟡中

**缓解措施**: 
- 后续可添加数据库支持
- 或使用LocalStorage持久化对话记录

## 🆕 RAG 3.0 升级经验（2026-07-10）

### 有效模式

#### 5. RAG 混合检索 + 多模块管道架构

**模式**: 检索管线从单一向量检索升级为 4 模块管道（QueryExpand → Hybrid → Validate → Memory），每个模块可独立开关/降级，不阻塞主流程。

**适用场景**: 需要同时支持精确关键词匹配和语义理解的检索场景。

**收益**: 
- BM25+Dense 混合检索覆盖精确匹配和语义理解
- 内容校验过滤无关噪音，提高回答质量
- 多轮记忆减少重复检索，响应更快

#### 6. Docker 容器依赖管理：快照 + 条件回退

**模式**: 导出 site-packages 全量快照（tar.gz），Dockerfile 条件解压。有快照则秒级恢复，无快照则 pip install 自动降级。

**适用场景**: Python Docker 镜像构建，避免大依赖（PyTorch）重复下载。

**收益**: 
- 有快照时构建从分钟级降到秒级
- 无快照时自动降级，不阻塞构建
- pip 全局镜像源（清华源），所有安装自动走国内加速

### 踩过的坑

#### 4. macOS wheels 与 Linux Docker 不兼容

**问题**: 在 Mac 宿主机执行 `pip download` 下载的 wheels 文件（.whl）带有 `macosx` 平台标记，放入 Docker Linux 容器后无法安装。

**严重程度**: 🔴高

**根因**: `pip download` 默认下载当前平台（macOS）的二进制包，与 Linux 架构不兼容。

**方案**: 
- 不要在宿主机 `pip download`，让 Docker 内部下载 Linux 版本
- 或指定 `--platform manylinux2014_x86_64`
- 最优方案：导出容器内已安装的 site-packages 作为快照

**影响面**: 所有需要在 Docker 中安装 Python 依赖的场景

**预防措施**: 永远从 Docker 容器内部安装/导出 Python 包，不跨平台混用。

#### 5. pydantic 版本与 Python 3.12 兼容性

**问题**: pydantic 2.6.1 + Python 3.12 触发 `ForwardRef._evaluate() missing 1 required keyword-only argument: 'recursive_guard'`，导致 langchain 导入崩溃。

**严重程度**: 🔴高

**根因**: langchain 依赖 langsmith，langsmith 内部使用 pydantic v1 兼容层，与 Python 3.12 不兼容。

**方案**: 升级 pydantic 到 2.13.4 修复兼容性。

**影响面**: 所有使用 langchain + Python 3.12 的项目

**预防措施**: 新项目固定 pydantic >= 2.10，老项目升级时注意测试 langchain 导入。
