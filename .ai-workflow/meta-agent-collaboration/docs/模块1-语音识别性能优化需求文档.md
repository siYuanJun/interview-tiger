## 三、需求范围与约束

### 纳入范围

| 编号 | 优化模块 | 目标 |
|:----:|---------|------|
| M1 | **断句优化 (B)** | 替换固定 1.5s 超时为自适应端点检测，减少误判断句 |
| M2 | **后处理管线 (C)** | 增加同音字纠错、术语修正、置信度过滤、LLM 辅助清洗 |

### 排除范围

| 项目 | 原因 |
|------|------|
| 替换 ASR 引擎（火山引擎/腾讯云 ASR 等） | 需企业认证或付费（R2、R5） |
| 前端 AudioContext 自定义采集 + 自研 VAD | 投入产出比低，Web Speech API 已内置基础 VAD |
| 离线本地 ASR（如 Whisper.cpp） | 浏览器端部署复杂度高，体积大，超出本次优化范围 |

### 硬约束

| 约束 | 来源 |
|------|:----:|
| 必须基于 Web Speech API，不能替换引擎 | R2、R5 |
| 必须免费，不引入付费服务 | R5 |
| 不能影响现有双向对话的实时性 | R4 |
| 断句延迟上限：用户说完后 1 秒内触发 LLM | 推导 |

## 四、子模块划分

### 4.1 M1：断句/端点检测优化

**关联痛点**：R7（语速快丢字）、场景双向对话（R4）

**关联代码**：
- `useSpeech.ts` 第 104-113 行（pauseTimer 固定超时）
- `useSpeech.ts` 第 80-123 行（onresult 回调）
- `InterviewPage.vue` 第 118 行（resumeListening 后立即重新开始）

**当前问题**：
1. 固定 1.5s 超时忽略说话人语速差异：慢速说话者会被过早切断，快速说话者断句延迟
2. `pauseRecognition()` 会调用 `recognition.stop()`，导致需要 `resumeListening()` 完全重启，引入额外延迟
3. 没有利用 `confidence` 值辅助断句决策

**优化方向（待 functional spec 细化）**：
- 自适应超时：根据单位时间内的字符输出速率动态调整超时阈值
- 语义完整性辅助：检查 finalText 是否以句末标点结束（。！？…）
- 双阈值策略：短超时做候选，长超时强制提交

### 4.2 M2：识别结果后处理管线

**关联痛点**：R6（同音字混淆）、R7（漏字多字）、R8（专业术语识别错误）

**关联代码**：
- `question_judge.py` 第 55-89 行（validate_question）
- `transcript.py` 第 39-88 行（process_transcript）
- `useSpeech.ts` 第 122 行（confidence 字段未使用）

**当前问题**：
1. 无同音字纠错机制，如 "热格" → 不会自动修正为 "RAG"
2. 无专业术语词库，面试场景高频术语（微服务、容器化、K8s）识别不准
3. `confidence` 字段被忽略，低置信度结果直接提交
4. 没有利用 LLM 做轻量文本清洗（项目已有 LLM 调用链路）

**优化方向（待 functional spec 细化）**：
- 面试领域术语白名单 + 拼音纠错
- confidence < 阈值的结果标记为"待确认"
- LLM 辅助纠错：提交前用轻量 prompt 做文本清洗
- 上下文连贯性：对比前后两次识别结果的字符级差异，修正跳变

### 4.3 M3：LLM 触发时机优化

**关联痛点**：R4（双向对话场景中 LLM 响应时机）

**关联代码**：
- `InterviewPage.vue` 第 101-178 行（handleSpeechResult 流程）
- `transcript.py` 第 39-88 行（process_transcript → 入库 → 返回）

**当前问题**：
1. `isFinal=true` → 立即 `submitTranscript()` → 立即 `resumeListening()`，无缓冲
2. 没有区分"用户对面试官说的话"和"用户向系统提问"，可能触发错误的 LLM 回答

**优化方向（待 functional spec 细化）**：
- 增加断句后的小缓冲队列（200ms），防止碎片化提交
- 提交前做意图识别：这条文本是"需要回答的问题"还是"对话中的闲聊"
